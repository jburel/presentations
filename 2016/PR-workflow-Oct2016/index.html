PR workflow, what can we improve?

October 11, 2016

(Ideal) Workflow

Goal: having such workflow as smooth and as quick as Possible

  • Open PR
  • Review
  • Merge

What's happening

  • PR open
  • Review:
    • Question, Test required, etc.
    • More change in code
    • More question, more tests
    • ...
  • Merge
    • In “limbo” state: Mark as "Ready to merge" but not merge yet, more unanswered questions, etc.

What can we improve: PR open

  • Improve PR template to indicate if UI/integration/unit tests (checklist)
  • CC relevant people e.g. @mtbc if DB change
  • Make sure tests are added/adjusted
    • Change to loading of data: integration/unit tests
  • Basically add anything that can help reviewers

What can we improve: Review

  • Functional review
  • Code review
  • Use github feature to approve changes e.g. omero-webtest PR 17
  • Everything is public: if discussion needed, better to schedule a zoom discussion than long debate.

What can we improve: Merge

  • Need clear statement, especially if there is a merge in the middle of discussion
  • Need both Functional/Code for PR to become mergeable
  • If PR mergeable and not merge yet, bring it back to attention. Do not wait to be blocked!
